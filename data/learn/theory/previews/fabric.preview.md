  - Lesson 2: What Is Microsoft Fabric? Microsoft Fabric is an end-to-end analytics platform delivered as Software-as-a-Service (SaaS). It was announced at Microsoft Build in May 2023 and reached general availability in November 2023. Since then it has become one of the fastest-growing data platforms in the enterprise market. The simplest way to understand Fabric: it is one platform that does everything your data team needs — data movement, data engineering, data warehousing, real-time analytics, data science, and business intelligence — without requiring you to provision, connect, or manage separate services. The SaaS Model and What It Changes Traditional analytics platforms are Platform-as-a-Service (PaaS). You provision a cluster, configure networking, manage scaling, apply patches, and monitor infrastructure. You are responsible for keeping the lights on. Fabric is SaaS. Microsoft manages the infrastructure entirely. There are no clusters to provision, no virtual networks to configure (unless you choose to), no patches to apply. You open a browser, sign in, and start working. This is not a cosmetic difference. It changes the economics and the operational model fundamentally: • No idle costs for paused clusters — Fabric’s Spark pools start on demand from warm pools and cost nothing when idle. • No infrastructure management — Microsoft handles scaling, patching, availability, and disaster recovery for the platform. • Unified billing — A single capacity unit (CU) pool serves all workloads. You do not get separate bills for storage, compute, and networking from seven different services. • Instant provisioning — Creating a Lakehouse, a Warehouse, or a Notebook takes seconds, not minutes or hours. • Automatic updates — New features appear in the platform without migration projects. You are always on the latest version. Positioning: Fabric vs. Databricks vs. Synapse vs. Snowflake The enterprise analytics market has several major platforms. Understanding where Fabric sits helps you evaluate it honestly: Platform Model Strengths Key Difference from Fabric Microsoft Fabric SaaS (unified) All-in-one: ETL, Spark, SQL, BI, streaming, ML in one portal with one lake Fabric IS Fabric Databricks PaaS (multi-cloud) Best-in-class Spark, MLflow, Unity Catalog, Delta Lake originator Separate platform; no native BI layer; you bring your own storage and orchestration Azure Synapse Analytics PaaS (Azure) SQL pools, Spark pools, pipelines in one Azure resource Fabric’s predecessor; being deprecated in favor of Fabric; no SaaS model Snowflake SaaS (multi-cloud) Elastic SQL warehouse, data sharing, strong governance SQL-centric; limited Spark/ML; no native BI; separate from orchestration (7 min)

  - Lesson 3: Delta Lake as the Default Format Every table in Fabric is a Delta Lake table. This is not optional. It is not a configuration choice. It is a platform-level decision that applies to every Lakehouse table, every Warehouse table, every SQL Database table that replicates to OneLake, and every mirrored table. Delta Lake is an open-source storage layer, originally created by Databricks, that adds three critical capabilities to plain Parquet files: ACID transactions, schema enforcement, and time travel. Understanding what Delta Lake is and how it works is essential because every table you touch in this course is a Delta table. What Makes a Delta Table A Delta table is not a single file. It is a directory containing two things: • Parquet data files — The actual data, stored in columnar Parquet format. Each file contains a subset of the rows. The number of files depends on how the data was written and how many partitions exist. • The _delta_log/ directory — A folder containing JSON files that form the transaction log. Each JSON file represents a commit — a record of what changed. The log is the source of truth for what data is currently in the table. (27 min)
  - Lesson 4: OneLake Shortcuts Shortcuts are virtual pointers to data that lives outside your Lakehouse — or even outside OneLake entirely. They make external data appear as if it were a native OneLake folder or table, without copying a single byte. This is one of Fabric’s most powerful architectural features. Enterprise data does not live in one place. It is scattered across Azure storage accounts, AWS S3 buckets, Google Cloud Storage, Dataverse environments, and other Fabric workspaces. Shortcuts let you bring all of that data into a single, unified namespace without data movement. How Shortcuts Work A shortcut is a metadata pointer stored in OneLake. When a query reads from a shortcut, the Fabric engine resolves the pointer and reads directly from the source. The data never transits through an intermediate copy. (24 min)

  - Lesson 9: Storage Metrics, Monitoring, and Costs OneLake storage is billed separately from compute. Understanding how storage is consumed, how to monitor it, and how to optimize costs is essential for production operations. How Storage Is Billed You pay for the total volume of data stored in OneLake, measured in GB per month. The pricing model is straightforward: • Data stored in OneLake — All Parquet data files, Delta transaction logs, and Files/ section contents count toward your storage consumption. • Shortcut data is NOT billed in OneLake — Because shortcuts point to external data, the storage cost stays with the source provider (ADLS Gen2, S3, GCS). • Mirrored data IS billed in OneLake — Because mirroring creates a physical copy in Delta format, the replicated data counts toward your OneLake storage. • Soft-deleted and versioned data — Old Delta file versions that have not been vacuumed still consume storage. This is a common source of unexpected costs.   - Lesson 2: What Is a Lakehouse? The Lakehouse is a Fabric item that provides a dual-interface storage experience: an unstructured file system (Files/) and a managed table system (Tables/), both backed by OneLake. It is the convergence point where raw data arrives and structured, analytics-ready data emerges. The Lakehouse Paradigm The lakehouse paradigm was popularized by Databricks and has become the dominant architectural pattern for modern data platforms. The core idea: use an open file format (Delta Lake on Parquet) stored in a data lake, but layer on the governance features traditionally found only in warehouses — ACID transactions, schema enforcement, SQL access, and fine-grained security. Fabric’s Lakehouse is a concrete implementation of this paradigm. When you create a Lakehouse in Fabric, you get: • A Files/ section — A folder hierarchy in OneLake for landing raw, unstructured, or semi-structured data. CSVs, JSONs, images, Parquet files, XML — anything can go here. No schema is enforced. This is your landing zone. • A Tables/ section — A managed area where Delta tables live. Every table here has a schema, ACID transactions, and a transaction log. These tables are queryable by Spark, by the SQL analytics endpoint, and by Power BI. • A SQL analytics endpoint — An auto-generated, read-only T-SQL interface that exposes every Delta table in the Tables/ section. SQL analysts can query Lakehouse tables without Spark, without Python, without notebooks. They just write T-SQL. • A default semantic model — An auto-generated Power BI semantic model that makes Lakehouse tables immediately available for reporting. You can build a Power BI report directly on top of a Lakehouse without any additional modeling. (12 min)

  - Lesson 8: Lakehouse vs. Warehouse — When to Use Which This is one of the most common questions in Fabric: should I use a Lakehouse or a Warehouse?   - Lesson 9: Lakehouse Best Practices A well-organized Lakehouse is the difference between a reliable data platform and a data swamp. These best practices apply to every Lakehouse you build. File Organization • Keep the Files/ section organized with a clear folder hierarchy: raw/, staging/, archive/. • Use date-based subfolders for incremental data (e.g., raw/orders/2026-01-15/). • Never leave raw files in Files/ indefinitely — process them into Tables/ and archive or delete the originals. • Avoid deeply nested folder hierarchies. Two to three levels of depth is sufficient. Table Organization • Name tables descriptively: orders, dim_customers, fact_daily_revenue — not table1, temp, or test_final_v2. • Use the medallion pattern (Bronze, Silver, Gold) to organize tables by data quality level. Different Lakehouses for each layer, or clear naming prefixes (bronze_orders, silver_orders, gold_fact_orders). • Document table purposes in the OneLake Catalog using descriptions and tags. • Endorse (Promote or Certify) tables that are ready for downstream consumption. Table Maintenance Delta tables require regular maintenance to stay performant: • OPTIMIZE weekly (or daily for high-volume tables) to compact small files. Target file sizes of 128 MB to 1 GB. • VACUUM weekly after OPTIMIZE to reclaim storage from old file versions. Use the default 7-day retention. • Z-ORDER on columns that are frequently used in WHERE clauses. This clusters data for faster filtered reads. • Monitor table sizes with DESCRIBE DETAIL. Watch for tables with thousands of small files — this is the most common performance problem. (13 min)

  - Lesson 3: Copy Activity Deep Dive The Copy Activity is the most-used activity in Data Factory. It moves data from a source to a destination (called a “sink” in ADF terminology). In Fabric, the most common pattern is copying data from an external source into a Lakehouse. Source and Sink Configuration Every Copy Activity has a source connection and a sink connection. In Fabric, configuring these is simpler than in ADF because you reference Fabric items directly: • Source: Select from 300+ built-in connectors — Azure SQL Database, Blob Storage, SharePoint, REST APIs, Salesforce, Snowflake, Oracle, SAP, and hundreds more. Each connector has specific configuration options (e.g., query for SQL sources, file path for storage sources). • Sink (Destination): Select a Fabric item — typically a Lakehouse (Files/ or Tables/ section), a Warehouse table, or a KQL Database. For Lakehouse targets, you choose whether to land in Files/ (as-is file copy) or Tables/ (auto-converted to Delta). Column Mapping The Copy Activity can map source columns to destination columns. This is useful when: • Source and destination have different column names. • You want to drop columns during copy (projection). • You need to change data types (e.g., source sends string dates, destination expects DATE type). For simple copies where source and destination schemas match, auto-mapping handles the mapping automatically. For schema drift scenarios (where source columns may appear or disappear), enable “Allow schema drift” in the sink settings. Performance Tuning Copy Activity performance depends on three factors: • Parallelism (Data Integration Units / DIUs): The number of parallel threads used for the copy. Higher DIU counts = faster copies for large datasets. Start with Auto and increase manually if throughput is insufficient. • Source partitioning: For SQL sources, enable parallel reads by partitioning the source query (e.g., by a date or ID range). This reads multiple chunks simultaneously instead of a single sequential scan. • File format and compression: Parquet with Snappy compression is the optimal format for OneLake ingestion. It provides the best combination of read performance and storage efficiency. Avoid landing as CSV if possible.   - Lesson 3: Capacity Management and Monitoring All Fabric workloads consume Capacity Units (CUs) from a shared capacity pool. Understanding how CU consumption works is essential for cost management and performance reliability. How Capacity Works • Capacity SKUs: Fabric capacities are purchased in SKU sizes (F2, F4, F8, F16, F32, F64, F128, etc.). Each SKU provides a defined number of CUs per second. Higher SKUs provide more parallel compute. • Shared pool: All workloads (Spark, Warehouse, Power BI, Pipelines, KQL) draw from the same CU pool. A large Spark job and a Warehouse query compete for the same CUs. • Throttling: When CU demand exceeds capacity, Fabric throttles by queuing requests. Interactive workloads (report queries) are prioritized over background workloads (Spark jobs, pipeline runs). • Smoothing: CU consumption is smoothed over time windows. Short bursts are tolerated; sustained overconsumption triggers throttling. Fabric smoothes background operations over a 24-hour window and interactive operations over shorter windows.
